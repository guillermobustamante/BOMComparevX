# Sprint S12 - Results Revision Chain

## 1. Sprint metadata
- Sprint: `S12`
- Theme: `Results Revision Chain`
- Scope type: `Frontend workflow extension with targeted backend contract support`
- Owner: `Product + Engineering`
- Status: `Completed`

## 2. Sprint goal
Extend the approved Antigravity Results workspace so a user can continue a BOM session by uploading one new file, automatically comparing it against the latest file in that same session, and reviewing prior comparisons from inside the Results route.

## 3. Locked UX decision set
- Chained upload always compares `latest file vs new file`.
- No baseline chooser is shown in Sprint 12.
- The chained upload starts from a modal on `/results`.
- The modal supports drag and drop and a `Select file` action.
- After upload intake succeeds, the user is redirected immediately into the new in-progress Results workspace.
- `/results` must expose a `Previous comparisons` action for the current session.
- The previous-comparisons view is modal-based and session-scoped.
- Each comparison entry shows:
  - comparison label
  - upload date/time
  - user
  - latest state or historical timestamp treatment
  - open action
- Labels are system-generated by default and can be renamed.
- Previous comparisons are not read-only; rename/open/delete remain available.
- Exports and sharing apply only to the currently open comparison.
- Validation/comparison issues remain popup-based and do not navigate away on failure.
- All uploaded files in a session represent the same BOM lineage.

## 5. Sprint 12 user stories

### S12-01 - Chained upload from Results
As a user reviewing a comparison, I want to upload one new BOM file from the Results page so that I can continue the current session without going back to the original intake screen.

Status:
- `Completed`

### S12-02 - Latest-vs-new comparison rule
As a user working through revision history, I want the system to always compare the latest file in the session against the new upload so that the comparison chain stays deterministic and easy to understand.

Status:
- `Completed`

### S12-03 - Immediate redirect into the next in-progress Results workspace
As a user launching the next comparison, I want the system to take me directly into the new running Results workspace so that I stay in the active review context.

Status:
- `Completed`

### S12-04 - Previous comparisons modal
As a user reviewing a BOM lineage, I want a session-scoped previous-comparisons modal so that I can inspect and reopen prior comparison steps without leaving Results.

Status:
- `Completed`

### S12-05 - Rename support for session comparisons
As a user collaborating across Engineering, Purchasing, Sales, and support roles, I want system-generated comparison labels that can be renamed so that each comparison step can be given business-meaningful context.

Status:
- `Completed`

### S12-06 - Table-first Results polish
As a user spending most of my time on the parts list, I want status chrome to stay compact so that the comparison table remains the dominant element on the page.

Status:
- `Completed`

## 6. Implementation plan executed
- Extend upload validation/intake to support `sessionId + one new file` chained intake.
- Reuse the latest stored right-side revision in the session as the next comparison baseline.
- Keep the same `sessionId` across the revision chain.
- Expose session-filtered comparison history through the history API with revision-pair metadata and generated labels.
- Add Results toolbar actions for:
  - upload next revision
  - previous comparisons
- Add Results modal for chained upload with drag/drop and file picker.
- Redirect to `/results?sessionId=...&leftRevisionId=...&rightRevisionId=...` after chained intake acceptance.
- Add Results previous-comparisons modal with rename, open, and delete actions.
- Keep history/open behavior aligned by opening stored revision pairs instead of raw upload job IDs.

## 7. Scope summary
- Backend contract support for one-file chained intake in an existing session
- Results modal workflow for chained revision upload
- Results previous-comparisons modal
- History enrichment for session-specific comparison reopening
- Progress-indicator polish for the Results toolbar

## 8. Non-goals
- Baseline chooser / branching comparison graphs
- Cross-session comparison selection
- New backend domain expansion beyond session-chain support
- Persistent comparison-label storage beyond existing history rename behavior

## 9. Residual notes
- Session-chain reopening relies on stored revision-pair availability in the running environment.
- Authenticated-route latency remains a separate carry-forward optimization item from Sprint 11.
